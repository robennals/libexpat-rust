# Plan: Encoding Architecture

## Problem

~13 tests fail due to encoding-related issues:
- 4 UTF-16 tests (attribute, PE, bad attr keyword, bad doctype)
- 4 Latin-1/UTF-16 BOM tests (ext entity with BOM switch)
- 2 BOM/encoding tests (ext entity set encoding, ext entity set BOM, external BOM consumed)
- 3 unknown encoding tests (success, user data, bad ignore)

The root cause is an architectural divergence: C tokenizes in the document's native encoding using encoding-specific tokenizer functions, while our Rust code transcodes everything to UTF-8 upfront in `parse()` and tokenizes only UTF-8.

## How C Does It

### Encoding struct (xmltok.c)
C has an `ENCODING` struct with function pointers:
```c
struct encoding {
    SCANNER scanners[NUM_STATES];  // content_tok, prolog_tok, etc.
    int minBytesPerChar;           // 1 for UTF-8, 2 for UTF-16
    char isUtf8, isUtf16;
    // ... byte type tables, converter function
};
```

Pre-built encodings: `utf8_encoding`, `internal_utf8_encoding`, `little2_encoding` (UTF-16LE), `big2_encoding` (UTF-16BE), `latin1_encoding` (ISO-8859-1).

### Encoding detection flow
1. `prologInitProcessor` calls `initializeEncoding` → `XmlInitEncoding` → sets up `m_initEncoding` (a wrapper that auto-detects)
2. First call to tokenizer: `initScan` examines BOM/byte patterns, sets `*encPtr` to the correct encoding struct, returns a token
3. From then on, `m_encoding` points to the real encoding and the tokenizer uses encoding-specific byte type tables
4. If `processXmlDecl` finds an encoding declaration, it may switch `m_encoding` to a different encoding
5. `handleUnknownEncoding`: user callback fills a 256-int map; `XmlInitUnknownEncoding` builds a custom encoding from that map

### Key: tokenizer is encoding-aware
The UTF-16 tokenizers (`little2_contentTok`, `big2_prologTok`, etc.) work directly on 2-byte characters. They don't transcode. The byte type tables map byte patterns to character classes. This means UTF-16 data is tokenized in-place without conversion.

### For external entities
Child parsers created with `XML_ExternalEntityParserCreate` go through their own encoding detection:
- `externalEntityInitProcessor` → `externalEntityInitProcessor2` → `externalEntityInitProcessor3`
- This detects BOM in the child parser's data stream, sets the child's encoding
- The child may have a different encoding than the parent

## How Rust Currently Does It

### parse() transcodes upfront
In `parse()`, before any tokenization:
1. `detect_and_transcode()` → `init_scan()` detects encoding → transcodes UTF-16 to UTF-8
2. Transcoded UTF-8 data is stored in `self.buffer`
3. All tokenizers (`content_tok`, `prolog_tok`) only handle UTF-8

### Problems with this approach
1. **Byte-at-a-time UTF-16**: If a user feeds 1 byte at a time (as `_XML_Parse_SINGLE_BYTES` does in tests), we may get half a UTF-16 code unit. `detect_and_transcode` handles this with `pending_utf16_byte`, but it's fragile.
2. **BOM in child parsers**: External entity child parsers run their own `parse()`, which runs `detect_and_transcode`. But BOM detection in child parsers is currently incomplete — `externalEntityInitProcessor2` is missing.
3. **Unknown encoding**: The handler is called and we get the map, but we only use it for transcoding in `transcode_custom_encoding`. We don't create an encoding-aware tokenizer.
4. **Encoding switch mid-stream**: If `processXmlDecl` detects a different encoding, we can't switch tokenizers because everything is already UTF-8.
5. **Position tracking**: C tracks byte positions in the original encoding. Our transcoding loses this correspondence.

## Available Approaches

### Approach A: Multi-encoding tokenizer (match C architecture)

Port C's encoding architecture: implement UTF-16LE, UTF-16BE, and Latin-1 tokenizers that work on raw bytes. The `Encoding` trait would have implementations for each encoding. `initializeEncoding` sets the active encoding. No upfront transcoding.

**Pros:**
- Exact C match — easiest to verify all edge cases
- Handles byte-at-a-time feeding naturally
- Position tracking is trivially correct
- No transcoding overhead

**Cons:**
- Massive amount of code: each encoding needs its own `content_tok`, `prolog_tok`, `cdataSectionTok`, etc.
- In C these are generated by macros (`#define BYTE_TYPE(enc, p) ...`). Rust doesn't have this — we'd need either generics, trait implementations, or code duplication.
- The UTF-16 tokenizers are ~2000 lines of macro-expanded C
- Risk of introducing bugs in new tokenizer code

**Complexity:** ~2000+ lines of new tokenizer code. Weeks of work.

### Approach B: Transcode to UTF-8 at chunk boundaries (current approach, improved)

Keep the transcode-to-UTF-8 approach but fix the edge cases:
1. Fix byte-at-a-time UTF-16 by buffering in `parse()` until we have complete code units
2. Port `externalEntityInitProcessor2` for BOM detection in child parsers
3. Port `handleUnknownEncoding` to create a transcoding function from the map
4. Handle encoding switch in `processXmlDecl` by re-transcoding remaining data

**Pros:**
- Builds on existing code
- Single tokenizer (UTF-8) — less code, fewer bugs
- Rust's `String` type assumes UTF-8, so all string handling is natural
- Memory-safe by construction (no byte-type tables to get wrong)

**Cons:**
- Transcoding has CPU and memory overhead (copy + convert)
- Byte-at-a-time buffering adds complexity
- Position tracking requires maintaining original↔transcoded position map
- Not a 1:1 match with C — harder to verify via AST comparison

**Complexity:** ~200 lines of changes to existing code.

### Approach C: Transcode at the tokenizer boundary (hybrid)

Keep UTF-8-only tokenizers but transcode inside the tokenizer call, not in `parse()`. Each tokenizer call would:
1. Accept raw bytes + encoding enum
2. If encoding is UTF-8, tokenize directly
3. If encoding is UTF-16/Latin-1, transcode the current token's bytes to UTF-8, then tokenize
4. Return token + number of raw bytes consumed (for position tracking)

**Pros:**
- Single tokenizer implementation (UTF-8)
- Encoding-aware at the right level (tokenizer, not parser)
- Position tracking is per-token, not per-chunk
- Handles byte-at-a-time: tokenizer says "need more bytes" naturally
- No upfront bulk transcoding — memory efficient

**Cons:**
- Still differs from C architecture
- Per-token transcoding may be slower than bulk
- Tokenizer interface changes (now takes encoding parameter)

**Complexity:** ~400 lines of changes.

## Recommendation: Approach B (improve current transcoding)

Approach B is recommended because:
1. **It works for most cases already** — 255 of 290 tests pass
2. **The remaining failures have specific causes** that can be fixed individually
3. **Approach A is extremely high effort** for marginal benefit
4. **Safety** — the existing UTF-8 tokenizer is well-tested

### Specific fixes needed

#### Fix 1: Port externalEntityInitProcessor2 (fixes BOM tests)
C's `externalEntityInitProcessor2` handles BOM detection in external entity child parsers. Port this to Rust:
- Check first 2-4 bytes for BOM (FF FE, FE FF, EF BB BF)
- Skip BOM bytes and set encoding for the child parser
- Then call the content processor

This should fix: `test_ext_entity_set_bom`, `test_external_bom_consumed`, `test_ext_entity_latin1_utf16*_bom` (4 tests)

#### Fix 2: Port handleUnknownEncoding (fixes unknown encoding tests)
Currently we call the user's handler and get the 256-int map, but we only validate it. We need to:
1. Build a transcoding function from the map (map[byte] → unicode codepoint)
2. Store it as the parser's custom encoding
3. Use it to transcode subsequent data to UTF-8

This should fix: `test_unknown_encoding_success`, `test_unknown_encoding_user_data_secondary`, `test_unknown_encoding_bad_ignore` (3 tests)

#### Fix 3: UTF-16 byte-at-a-time buffering (fixes UTF-16 tests)
The test harness feeds data one byte at a time. For UTF-16, we need at least 2 bytes per character. Our `pending_utf16_byte` mechanism handles this for data after BOM detection, but the BOM detection itself may fail with 1 byte.

The fix: ensure `detect_and_transcode` properly buffers initial bytes when UTF-16 encoding is protocol-specified (not just BOM-detected).

This should fix: `test_utf16_attribute`, `test_utf16_pe`, `test_bad_attr_desc_keyword_utf16`, `test_bad_doctype_utf16` (4 tests)

#### Fix 4: Encoding switch via processXmlDecl
When an XML declaration specifies a different encoding than auto-detected, we need to:
1. Re-encode any buffered data that was transcoded with the wrong encoding
2. Switch the transcoding function for subsequent chunks

Currently inlined in `handle_prolog_role`'s XmlDecl case. Should be extracted to `process_xml_decl` function matching C.

This may fix: `test_ext_entity_set_encoding`, `test_bad_doctype` (2 tests)

#### Fix 5: UTF-16 entity tests
`test_entity_public_utf16_be/le` — these test PUBLIC entity declarations in UTF-16. Should work once UTF-16 byte-at-a-time is fixed.

### Encoding tests that will NOT be fixed by Approach B
If any test requires in-place UTF-16 tokenization (not just transcoding), it would need Approach A or C. Based on analysis, all current failing tests should be fixable with improved transcoding.

## Implementation Order

1. **Fix 1: externalEntityInitProcessor2** — most tests fixed per effort
2. **Fix 2: handleUnknownEncoding** — straightforward, 3 tests
3. **Fix 3: UTF-16 byte-at-a-time** — may be partially working already
4. **Fix 4: processXmlDecl extraction** — cleanup, enables proper encoding switch
5. **Fix 5: UTF-16 entity tests** — likely fixed by Fix 3

## If Approach B proves insufficient

If specific tests require true encoding-aware tokenization (not just transcoding), consider Approach C as a surgical upgrade: add encoding awareness to individual tokenizer functions that need it, rather than rewriting all tokenizers.
