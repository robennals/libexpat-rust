# C-to-Rust Porting Process for xmlparse.c

## Goal
Port libexpat's xmlparse.c to idiomatic Rust with exact behavioral equivalence,
verified by comparison tests against the real C library (via FFI).

## Current State (2026-03-21)
- **Tests**: 124 passing, ~63 failing, ~289 ignored
- **Core loop**: `doContent`, `doProlog`, `epilogProcessor`, `doCdataSection` exist but have gaps
- **doProlog**: Only 8 of 80 role cases implemented (the role handler is the main gap)
- **Key missing features**: namespace processing, DTD entity resolution, external entities

## Tools

All tools are in `scripts/`. Run from project root.

### AST Structural Comparison (primary tool)
```bash
# Compare a specific C↔Rust function pair
python3 scripts/ast-compare.py doContent do_content

# Compare ALL known pairs — shows all divergences
python3 scripts/ast-compare.py --all

# List switch/match cases in a function
python3 scripts/ast-compare.py --list-cases doContent c
python3 scripts/ast-compare.py --list-cases do_content rust
```

The output shows:
- **missing_match_arms**: Token/role cases C handles but Rust doesn't
- **missing_errors**: Error codes C returns that Rust doesn't
- **missing_handlers**: Handler calls C makes that Rust doesn't
- Per-case divergences (which specific case is missing what)

### Porting Status & Call Tree
```bash
# Show functions ready to port (all deps already ported)
python3 scripts/port-function.py ready

# Show call tree and status of core functions
python3 scripts/port-function.py analyze

# Extract a C function for reading
python3 scripts/port-function.py extract doContent

# Suggest how to decompose a large function
```

### Running Tests
```bash
cd expat-rust

# Run all tests (fast, <10s)
RUST_TEST_THREADS=1 cargo test

# Run one test file
RUST_TEST_THREADS=1 cargo test --test c_comparison_tests

# Run one specific test
RUST_TEST_THREADS=1 cargo test --test basic_tests_0 test_nul_byte -- --exact

# Show per-file results
for f in basic_tests_0 basic_tests_2 basic_tests_3 basic_tests_missing c_comparison_tests misc_tests ns_tests; do
  result=$(RUST_TEST_THREADS=1 cargo test --test "$f" 2>&1 | grep "test result:")

done
```

## Process: How to Port

### Step 1: Identify what to work on
Run `python3 scripts/ast-compare.py --all` to see all divergences.
Pick the divergence with the highest test impact.

Alternatively, run `python3 scripts/port-function.py ready` to find
leaf functions that can be ported next.

### Step 2: Check the C source
```bash
python3 scripts/port-function.py extract <c_function_name>
```
Read the C code. Understand what it does structurally.

### Step 3: Port bottom-up
**Rule: Only port a function after all functions it calls are solid.**

For large functions (>200 lines), decompose into sub-functions:
```bash
python3 scripts/port-function.py decompose <c_function_name>
```
The structural comparison tool allows Rust to have intermediate functions
that C doesn't have, as long as the overall behavior matches.

### Step 4: Verify with AST comparison
```bash
python3 scripts/ast-compare.py <c_func> <rust_func>
```
Check that all cases, errors, and handlers match.

### Step 5: Run tests
```bash
cd expat-rust && RUST_TEST_THREADS=1 cargo test
```
Check that test count went up (or at least didn't go down).

### Step 6: Commit incrementally
Each fix should be a small commit with the test count in the message.

## Architecture

### File Layout
```
expat-rust/src/
  xmlparse.rs    — Main parser (the file being ported from xmlparse.c)
  xmltok.rs      — Token types, encoding traits, XML declaration parsing
  xmltok_impl.rs — Tokenizer implementation (content_tok, prolog_tok, etc.)
  xmlrole.rs     — Prolog role state machine (maps tokens to semantic roles)
  siphash.rs     — Hash function for DoS protection
  char_tables.rs — Character classification tables
  nametab.rs     — Name character tables
```

### How Parsing Works
1. `parse()` stores data in buffer, calls `run_processor()`
2. `run_processor()` dispatches to current processor:
   - `prolog_processor()` → `do_prolog()` — handles XML declaration, DOCTYPE, etc.
   - `content_processor()` → `do_content()` — handles elements, text, entities
   - `epilog_processor()` — handles post-root-element content
3. Each processor calls tokenizer functions from `xmltok_impl`:
   - `prolog_tok()` for prolog tokens
   - `content_tok()` for content tokens
4. `do_prolog()` additionally uses `xmlrole::xml_token_role()` to map
   tokens to semantic roles (DOCTYPE_NAME, ENTITY_VALUE, etc.)

### Key Patterns
- **C's switch on token type** → **Rust's match on XmlTok enum**
- **C's switch on role** → **Rust's match in handle_prolog_role()**
- **C's parser->m_handler** → **Rust's self.handler_name**
- **C's XML_ERROR_X** → **Rust's XmlError::X**
- **C's pool/hash** → **Rust's String/Vec/HashMap** (don't port C data structures)

### Role Handler Architecture (doProlog)
`do_prolog()` has a loop that:
1. Calls `prolog_tok()` to get the next token
2. Calls `xml_token_role()` to get the semantic role
3. Dispatches to `handle_prolog_role()` which has a match on Role

Currently 8 of 80 role cases are implemented. Each role case is a small
self-contained unit that can be ported independently. Treat each role case
like a leaf function.

## Key Principles

1. **Match C behavior exactly** — The C library (via expat-sys FFI) is ground truth
2. **Bottom-up** — Port leaves before callers
3. **AST comparison drives the work** — Don't guess, run the tool
4. **Incremental** — Each change should improve or maintain test count
5. **No ad-hoc fixes** — Always check C source when fixing divergences
6. **Safety guards** — do_content has iteration limits to prevent infinite loops

## What NOT to Do

- Don't port C memory management (pools, hash tables) — use Rust std
- Don't port accounting/billion-laughs — add later
- Don't port external entity processors until internal entities work
- Don't guess at behavior — run the C library via `expat-sys`
- Don't edit xmlparse.rs with parallel agents (they clobber each other)
- Don't run tests without `RUST_TEST_THREADS=1` (resource contention)
